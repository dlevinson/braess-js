Braess Explorer

Model documentation

This app is a browser-only research toy for exploring how a new link can change static user equilibrium, and how elastic demand converts higher generalized cost into lower equilibrium trips. It is deliberately transparent: one OD pair, small directed networks, visible cost functions, and numerical methods simple enough to inspect.

Network

Network and candidate link

The app solves a directed road network with one origin zone and one destination zone. The built-in preset is the classic four-node Braess topology: Origin, Upper, Lower, and Destination. The candidate link is the middle directed link from Upper to Lower.

Each comparison is solved twice. In the without added link case, the candidate link is disabled. In the with added link case, the candidate link is enabled only if it has positive lanes and positive capacity. This is why the app can show the link geometrically while still treating it as closed or not added.

The map view can show either scenario. The mathematical comparison is not a visual toggle: OFF and ON are separately built networks, separately solved, and then compared.

Costs

Link performance functions

Link costs are deterministic functions of link flow. The app supports affine and BPR-style costs. The current solver treats the reported link cost as generalized cost, so the units are intentionally abstract: minutes, dollars, utility-scaled time, or toy units, as long as the same units are used consistently. 

Affine: t_l(x_l) = a_l + b_l x_l

BPR:    t_l(x_l) = t0_l * (1 + alpha_l * (x_l / capacity_l)^beta_l)

For BPR links, t0 comes from a positive cost-a override when one is supplied. Otherwise it is computed as length / free_speed. For affine links, length does not automatically enter the cost unless the user edits the cost parameters to make it do so.

UI field Meaning
a / t0 override Affine intercept, or BPR free-flow-time override if the BPR model is selected.
b Affine congestion slope. Higher values make cost rise faster with flow.
capacity Used directly in BPR costs and also determines whether a link is structurally open.
lanes Determines whether a link is structurally open. A link with zero lanes is closed.

Equilibrium

Fixed-demand user equilibrium

In fixed-demand mode, demand is held at Initial trips. The app enumerates all simple OD paths in the enabled scenario network and solves Wardrop user equilibrium using Method of Successive Averages. 

Start with zero path flows.
For iteration k = 1, 2, ...
  1. Convert path flows into link flows.
  2. Evaluate link costs from current link flows.
  3. Find the shortest currently available OD path.
  4. Assign all demand to that shortest path.
  5. Average the assignment into the current path flows:
       f_path(k) = f_path(k-1) + (1/k) * (aon_path - f_path(k-1))

The convergence chart reports the relative gap: the flow-weighted excess path cost above the current shortest path cost, divided by demand times shortest-path cost. The solver stops after at least five iterations once the gap is below tolerance, or after the maximum iteration count.

Equilibrium generalized cost is computed as the demand-weighted average path cost in the final path-flow vector. For a well-converged user equilibrium, used paths should have nearly equal costs.

Demand response

Elastic-demand equilibrium

Elastic demand uses a linear inverse-demand curve: 

C(q) = A - Bq

Here A is the willingness-to-pay intercept, B is the slope, and A / B is the zero-cost X-intercept. The app labels that X-intercept as Initial trips, because it is the demand scale shown in the demand-curve editor. In fixed-demand mode, Initial trips is the actual fixed quantity. In elastic mode, Initial trips is the demand-curve intercept, not necessarily the equilibrium demand.

The elastic solver wraps the fixed-demand solver in an outer bisection search on q. For each trial quantity, it solves fixed-demand equilibrium and compares the network equilibrium cost with the inverse-demand willingness to pay. 

Find q such that:
  equilibrium network cost at q = A - Bq

If the no-flow OD travel cost is already greater than or equal to A, the app reports zero elastic demand. That means the demand curve chokes off before any positive trips are willing to use the network.

Outputs

Metrics and classification flags

Equilibrium cost

The average generalized cost experienced by equilibrium users in the scenario.

Demand

Fixed trips in fixed-demand mode, or the solved quantity where willingness to pay equals cost in elastic mode.

Total travel time

The sum over enabled links of flow times link cost.

Average travel time

Total travel time divided by demand, falling back to equilibrium cost when demand is zero.

Total system travel time = sum_l x_l t_l(x_l)

Consumer surplus = integral from 0 to q of (A - Bz) dz - C_eq q
                 = A q - 0.5 B q^2 - C_eq q

Total welfare = consumer surplus - total system travel time

Consumer surplus is only reported for elastic-demand runs. In fixed-demand mode it is set to zero because the current model is not using inverse demand to choose quantity.

Flag Definition
Braess paradox cost_on - cost_off > tolerance
Reduced demand cost_on > cost_off and demand_on < demand_off, both by tolerance.
Welfare loss welfare_on - welfare_off < -tolerance
Consumer-surplus loss consumer_surplus_on - consumer_surplus_off < -tolerance

The reduced-demand interpretation is intentionally narrow. It is not asking whether elastic demand is lower than fixed demand. It asks whether the added-link scenario carries fewer trips than the no-link scenario under the same elastic-demand curve. This is induced demand running in reverse: a network change raises equilibrium cost, so the elastic equilibrium moves to a lower quantity.

Preset

Default Braess case

The default network is calibrated so the fixed-demand Braess paradox is visible and the default elastic-demand case also produces reduced demand.

Component Default
Variable links Origin to Upper and Lower to Destination: t(x) = 10x
Constant links Origin to Lower and Upper to Destination: t(x) = 45
Candidate middle link Upper to Lower: t(x) = 0 when structurally open and included in the ON scenario.
Fixed demand Initial trips = 6 normalized toy demand packets.
Elastic demand A = 180, B = 30, so the zero-cost intercept is A / B = 6.

With these defaults, the fixed-demand run has the classic structure: the added middle link attracts users into individually rational route choices that raise equilibrium cost. Under the default elastic curve, that higher ON cost also lowers equilibrium demand relative to the OFF case.

Parameter space

Sweeps and outcome maps

The 2D sweep solves the OFF and ON scenarios for each cell in a user-defined parameter grid. The app stores one row per cell with OFF demand, ON demand, OFF cost, ON cost, deltas, and classification flags. The heatmap can show binary regions such as Braess paradox or continuous deltas such as change in cost or change in demand.

The elastic outcome map focuses on demand assumptions. It varies A and B, then optionally varies a third parameter Z, such as variable-link coefficient b, constant-link cost, or candidate-link cost. Each A-B-Z point is classified by Braess status and demand response.

Class Meaning Shown?
BP / reduced demand The added link worsens cost and lowers elastic demand. Yes
BP / no demand change The added link worsens cost but demand is equal within tolerance. Hidden in the 3D legend when it has zero count.
BP / increased demand Generally inconsistent with a downward-sloping inverse demand curve when cost rises. Logically prohibited for normal positive-slope demand response, so hidden when absent.
No BP / reduced demand Generally inconsistent with lower-or-equal ON cost under the same downward-sloping demand curve. Logically prohibited for normal positive-slope demand response, so hidden when absent.
No BP / no demand change The added link does not worsen cost and demand is equal within tolerance. Yes
No BP / increased demand The added link does not worsen cost and elastic demand rises. Yes

The outcome maps are numerical experiments. Coarse grids can miss boundaries, and very large grids may be slow in Safari because each point solves both OFF and ON equilibria. Use smaller step counts while exploring, then refine once the interesting region is visible.

Editing

Network editing semantics

Network editing is direct and local. Clicking a link exposes that link's own lanes, capacity, length, free speed, cost-function type, and cost parameters. Clicking a node exposes node coordinates. Dragging a node changes the drawing geometry and updates connected link lengths in proportion to the change in screen-space distance from the original preset geometry.

The model does not pretend that drawing geometry alone is a complete traffic model. For affine costs, changing length changes displayed link length but does not change cost unless cost parameters are changed. For BPR costs, length can affect free-flow time when no positive t0 override is supplied.

Closing a link is structural: a link with zero lanes or zero capacity is disabled. The candidate link also requires the ON scenario to be active before it contributes to route choice.

Scope

Limitations and extension points

This app is not a production traffic assignment engine. The deliberate constraints keep the model inspectable: one OD pair, small directed networks, simple-path enumeration, static user equilibrium, and a single inverse demand model.

Natural next extensions are multiple OD pairs, social optimum comparison, marginal-cost tolling, constant elasticity demand, richer path displays, and larger-network algorithms that avoid enumerating all simple paths. Those should be added carefully, because the app's main research value is that users can see exactly which assumptions produce Braess paradox, induced demand, or reduced demand.
